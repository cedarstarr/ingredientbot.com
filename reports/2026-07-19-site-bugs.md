## Categories audited — all clean

- **Error boundaries** — Complete coverage: `src/app/error.tsx`, `global-error.tsx`, `not-found.tsx`, plus per-route-group boundaries for `(app)`, `(admin)`, `(auth)`. Each uses the project style (client component, `useEffect` → `console.error` for monitoring, `AlertCircle`, dark-mode-aware Tailwind, reset + escape-hatch button).
- **Auth guards** — Global edge auth in `src/middleware.ts`: non-public API routes return 401, pages redirect to `/login`. Admin surfaces double-gated — middleware `isAdmin` check on `/admin` pages *and* explicit `session.user.isAdmin !== true` → 401 in both admin API routes (`admin/scripts/[name]/run`, `admin/trigger-cron`). Email-verification gate present.
- **API error handling** — Every route except the NextAuth catch-all handler wraps logic in `try/catch`. AI routes additionally rate-limit (`aiLimiter`), gate on `ANTHROPIC_API_KEY`/config, and return typed status codes (400/401/429/503/500).
- **AI-generated content rendering (security focus)** — **No unsanitized AI HTML.** Recipe suggestions are NDJSON parsed as data, never HTML. AI "Modified Recipe" output renders as React text children inside `<pre className="whitespace-pre-wrap">` (auto-escaped). The only two `dangerouslySetInnerHTML` uses are JSON-LD structured data wrapped in `safeJsonLdString()` (XSS-safe: escapes `&`/`<`, unit-tested in `src/lib/__tests__/utils.test.ts`).
- **Console statements** — No stray debug logs. All 5 non-error `console` calls are intentional (SW-registration warn, AI fallback warns, email-not-configured dev fallback).
- **Image handling** — No `<img>` tags; `next/image` used with `alt` present on every instance.
- **Empty states** — List clients (`saved-recipes-client`, `history-client`, `collections-client`) receive server-fetched props and each renders a `length === 0` empty state.
- **Dead links** — No `href="#"`, empty hrefs, or `TODO`/`FIXME` link stubs.
- **Loading states** — `(app)` and `(index)` route groups have `loading.tsx`; forms use per-action pending flags (`saving`, `isModifying`, `deletingId`, etc.).
- **Build warnings** — Only the Next 16 `middleware`→`proxy` convention-deprecation notice (see LOW-2).

## Deferred findings (MEDIUM+) → Linear

### MEDIUM-1 — Silent mutation failures on secondary actions (UX)
Several client components check `res.ok` (or omit the check) but give the user **no feedback when a mutation fails** — the action silently no-ops. Not a broken flow and not a security issue; all auth-critical, import, pantry, kitchen, and substitution flows *do* surface errors. Affects secondary/one-tap mutations only.

Components with a fetch mutation and no error feedback (`res.ok` false path is a silent no-op, or no catch):
- `src/components/recipe/collections-client.tsx` — `handleCreate` / `handleDelete` (lines 55–87): on non-ok, nothing happens.
- `src/components/recipe/delete-recipe-button.tsx`
- `src/components/recipe/recipe-rating.tsx`
- `src/components/recipe/recipe-tags.tsx`
- `src/components/recipe/collection-picker.tsx`
- `src/components/recipe/cooked-this-button.tsx`
- `src/components/recipe/share-recipe-button.tsx`
- `src/components/settings/dietary-profile-section.tsx`
- `src/components/meal-plan/meal-planner-client.tsx`

**Recommended fix:** add a `toast` (the project already uses a toast primitive in kitchen-panel/substitution-panel) on the `!res.ok` / `catch` branch of each mutation — e.g. `toast({ title: 'Could not save', variant: 'destructive' })`. Straightforward but touches ~9 files and warrants runtime verification against a running server, so it belongs in a dedicated pass rather than autonomous batch mode.

### LOW-1 — Missing `loading.tsx` in `(auth)` and `(admin)` route groups
`(app)` and `(index)` have loading skeletons; `(auth)` and `(admin)` do not. Auth pages are lightweight (low value), but `(admin)` pages run DB queries and would benefit from a skeleton. **Recommended:** add `src/app/(admin)/loading.tsx` matching the existing `(app)/loading.tsx` style.

### LOW-2 — Next 16 `middleware`→`proxy` deprecation warning
Build emits: `The "middleware" file convention is deprecated. Please use "proxy" instead.` This is a framework-convention rename tracked portfolio-wide (memory: `middleware.ts→src/proxy.ts`), not a site-specific QA bug. Out of scope for this pass; handle in the portfolio Next 16 migration.
