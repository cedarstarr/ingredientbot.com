## MEDIUM

### 1. Missing `@@index([site])` on `AICallLog`
**File:** `prisma/schema.prisma` — `AICallLog` model
**Issue:** The `site` column had no index. Any query filtering by site (e.g. per-site cost dashboards) caused a full table scan on `ai_call_logs`.
**Fix:** Added `@@index([site])` alongside the existing `feature` and `userId` indexes.

### 2. Broad `findUnique` in `/api/user/password` — full User row for password compare
**File:** `src/app/api/user/password/route.ts` line 25
**Issue:** `prisma.user.findUnique({ where: { id } })` without `select` fetched all 20+ User columns (session history, preferences, all flags) just to retrieve the `password` hash.
**Fix:** Added `select: { id, password, email, name }` — the only fields used by the password-change handler.

### 3. Broad `findUnique` in `/api/user/account` — full User row for account-delete confirm
**File:** `src/app/api/user/account/route.ts` line 28
**Issue:** Same pattern as above — full User fetch just to compare password before account deletion.
**Fix:** Added `select: { id, password, email, name }`.

## LOW

### 4. `meal-plan-digest` cron `findMany` without `take` cap
**File:** `src/app/api/cron/meal-plan-digest/route.ts` line 39
**Issue:** `prisma.mealPlan.findMany` was date-windowed but had no hard upper bound. A runaway batch (clock drift, schema rename of the date field) could scan the full table.
**Fix:** Added `take: 5000` safety cap with explanatory comment.
