## Phase 2 — Schema Audit

### HIGH — String enums (3 fields)

**`Recipe.difficulty` / `MealPlanSlot.mealType` / `User.chefPersonality`**

All three fields use `String` despite having a fixed, known set of values. String fields allow any value to be written — a typo or client bug can silently store invalid data that breaks filtering/display.

**Fix:** Added `Difficulty`, `MealType`, and `ChefPersonality` enum types to the schema. Updated all three fields to use the new enum types. Updated all callers with proper imports and casts.

### Findings that are NOT issues

- Foreign key indexes: all present (`@@index([userId])`, etc.)
- Float for money: no monetary values in schema
- Missing `updatedAt`: all models with `createdAt` also have `updatedAt` where appropriate
- Missing cascade rules: all FK `@relation` fields have explicit `onDelete`
- `AuditLog.userId`: nullable, append-only — intentionally no relation defined (preserves history on user delete)
- No models with 20+ fields
- No implicit many-to-many that should be explicit

## Phase 3 — Query Pattern Audit

### HIGH — Freemium gate missing on import route

**File:** `src/app/api/recipes/import/route.ts`

The `/api/recipes/import` save path created recipes without checking the freemium monthly limit. Free users could import unlimited recipes while the generation flow (`/api/recipes/cook`) enforced a 5/month cap. This is a business logic bypass.

**Fix:** Added the same freemium gate as `cook/route.ts`. Also wrapped the recipe create + user counter update in a `$transaction`.

### HIGH — Missing transaction: recipe create + counter update (`cook/route.ts`)

**File:** `src/app/api/recipes/cook/route.ts`

Two sequential writes without a transaction: `recipe.create` then `user.update` (recipeCount). If the second write failed (DB error, timeout), the recipe would exist but the counter wouldn't increment — allowing a free user to generate recipe N+1 without being charged for it.

**Fix:** Wrapped both writes in `prisma.$transaction([...])`.

### HIGH — PATCH /api/recipes/[id] accepts arbitrary body

**File:** `src/app/api/recipes/[id]/route.ts`

The PATCH handler passed the raw request body directly to `prisma.recipe.update({ data: body })`. Any field in the recipe model could be overwritten by a crafted request (e.g., `userId`, `isPublic`, `publicSlug`).

**Fix:** Allowlisted only `rating` (the only field actually PATCHed by the client). Added `select` to `findFirst` to avoid fetching the full record unnecessarily.

### MEDIUM — Missing transaction: user create + verificationToken create (`signup/route.ts`)

**File:** `src/app/api/auth/signup/route.ts`

Two sequential writes without a transaction: `user.create` then `verificationToken.create`. If the token write failed, the user account would exist but the email could never be verified — creating a permanently stuck unverified account.

**Fix:** Wrapped both writes in `prisma.$transaction([...])`.

### MEDIUM — Unbounded `findMany` on saved page

**File:** `src/app/(app)/saved/page.tsx`

`prisma.recipe.findMany` without `take` fetches all recipes for the user. A power user with thousands of recipes would cause a massive payload sent to the browser.

**Fix:** Added `take: 500`. Users with larger libraries use `/history` which has proper pagination.

### MEDIUM — GET /api/recipes/[id] returns full blobs

**File:** `src/app/api/recipes/[id]/route.ts`

`findFirst` without `select` returned all recipe fields including `rawText` (Text blob), `recipeData` (full JSON), `modifications` (JSON). Since no client currently calls this GET endpoint (recipe detail page fetches server-side), this was a latent issue but worth fixing.

**Fix:** Added `select` with only the lightweight fields needed.

### MEDIUM — adminScriptLog.findMany without select

**File:** `src/app/(admin)/admin/scripts/page.tsx`

`findMany` with `distinct` but no `select` fetched all columns of `AdminScriptLog`. Only 5 fields are used.

**Fix:** Added `select` with only the required fields.

## LOW — Items flagged, not fixed

### LOW — `allTagRows` full-table scan in history page

**File:** `src/app/(app)/history/page.tsx` (line 60)

Fetches all recipes for a user (only tags field) to build the distinct-tag filter list. For users with thousands of recipes, this adds a sequential query outside the `Promise.all`. The `select: { tags: true }` keeps it lightweight, but a `SELECT DISTINCT UNNEST(tags)` raw query would be more efficient at scale.

**Decision:** Not fixed. The tags column is a Postgres array type — no native Prisma equivalent for `DISTINCT UNNEST`. The fix would require a `$queryRaw`. At current scale this is LOW priority.

### LOW — Dashboard: allTimeCompletions outside Promise.all

**File:** `src/app/(app)/dashboard/page.tsx` (line 86)

A fifth query (`allTimeCompletions`) is fired sequentially after the `Promise.all([...])` block rather than being included in it. This adds one extra round-trip.

**Decision:** Not fixed. The query is simple (single select of `cookedAt` filtered by userId), and refactoring into the `Promise.all` would require restructuring the streak/heatmap logic. LOW priority.
