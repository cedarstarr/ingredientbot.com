## MEDIUM Issues (all fixed)

### M1 — Missing error boundaries in (app), (auth), (admin) route groups
**Files affected:** `src/app/(app)/`, `src/app/(auth)/`, `src/app/(admin)/`

Only `src/app/error.tsx` existed. Next.js error boundaries are scoped to their route segment; without segment-level `error.tsx` files, the global error boundary catches all errors but renders without any layout context (no AppNav, no admin sidebar). Users see a floating error card with no navigation.

**Fix:** Created three scoped error boundary files:
- `src/app/(app)/error.tsx` — links back to `/kitchen`
- `src/app/(auth)/error.tsx` — links back to `/login`
- `src/app/(admin)/error.tsx` — links back to `/admin`

---

### M2 — `console.log` in `src/lib/email.ts` logs email addresses to production server logs
**File:** `src/lib/email.ts` lines 28–30

When `ZEPTOMAIL_API_KEY` is not set, `sendEmail()` logs the recipient email address and subject to stdout. In production this would leak user PII into server logs.

**Fix:** Wrapped the fallback `console.log` calls in a `process.env.NODE_ENV !== 'production'` guard — dev fallback preserved, PII not logged in prod.

---

### M3 — `console.error(e)` in `kitchen-panel.tsx` exposes raw error objects client-side
**File:** `src/components/kitchen/kitchen-panel.tsx` lines 329, 390

`handleGenerate` and `handleImpressMe` caught fetch/stream errors and called `console.error(e)` with the raw exception. User-visible error state was already set correctly (`setError`), making the console call redundant and potentially exposing internal error details in browser consoles.

**Fix:** Removed the `console.error` calls. The `catch (e)` became `catch` (no binding needed).

---

### M4–M9 — API routes with no `try/catch` (unhandled DB errors would return Next.js default 500 with stack trace)
**Files:** 12 routes across `src/app/api/`

Routes that perform Prisma DB queries without wrapping them in try/catch will surface Next.js's default error response — which in some configs includes a stack trace. Covered:

| Route | Verbs wrapped |
|-------|--------------|
| `/api/collections` | GET, POST |
| `/api/collections/[id]` | PATCH, DELETE |
| `/api/user/dietary` | GET, PATCH |
| `/api/user/pantry` | GET, POST |
| `/api/user/pantry/[id]` | PATCH, DELETE |
| `/api/user/notifications` | GET, PATCH |
| `/api/user/usage` | GET |
| `/api/meal-plan` | GET, PUT |
| `/api/meal-plan/slot/[id]` | DELETE |
| `/api/recipes/[id]` | GET, PATCH, DELETE |
| `/api/recipes/[id]/tags` | PUT |
| `/api/recipes/[id]/collection` | PUT |
| `/api/recipes/[id]/cook` | POST |
| `/api/recipes/[id]/share` | POST, DELETE |
| `/api/auth/verify-email` | GET |
| `/api/auth/signout-all` | POST |

**Fix:** Added try/catch blocks returning `{ error: 'descriptive message' }` with 500 status.

Note: Routes already handled (`auth/signup`, `auth/forgot-password`, `auth/reset-password`, `recipes/cook`, `recipes/import`, `recipes/analyze-photo`, `recipes/generate`, `recipes/[id]/chat`, `recipes/[id]/substitute`, `recipes/[id]/nutrition`, `recipes/[id]/modify`, `recipes/[id]/convert-diet`, `admin/scripts/[name]/run`, cron routes, `user/email`, `user/account`, `user/password`, `user/kitchen-prefs`, `user/profile`) were already covered.

## LOW Issues (not fixed — acceptable)

### L1 — `console.error('[Error boundary]', error)` in `src/app/error.tsx`
Intentional: surfaces unhandled errors in monitoring tools (Sentry, etc.). Standard Next.js error boundary pattern.

### L2 — `console.error` in `src/lib/env.ts`
Intentional: config validation at startup — critical to log and fail fast when env vars are misconfigured.

### L3 — `console.error` in `src/app/api/user/email/route.ts`
Already inside a `try/catch` block with a proper 500 response. Used for server-side debugging of email change failures. Acceptable.

### L4 — `console.error` in cron routes (`welcome-drip`, `meal-plan-digest`)
Per-user delivery failure logging in a loop — acceptable operational pattern. Failures are per-iteration, not fatal to the cron run.
